# 온디맨드 분석 서비스에 배치 파이프라인(Airflow)을 별도로 둔 이유

## 배경

CAPD의 분석 리포트는 의사가 특정 환자의 리포트 화면을 열 때 그 자리에서 계산되는 온디맨드 방식으로 동작한다. 계산 결과는 캐시 테이블(Silver/Gold)에 저장되어 다음 조회부터는 재사용되지만, 이 캐시는 누군가 실제로 그 환자의 리포트를 한 번이라도 열어봐야만 채워진다는 한계가 있다. 즉 의사가 아직 열어보지 않은 환자는 이상치가 있어도 목록 화면에 배지가 뜨지 않는다 — 코드 버그가 아니라 온디맨드 구조 자체의 한계다.

이 한계를 해소하려면, 사람이 화면을 열길 기다리지 않고 전체 환자를 선제적으로 스캔해서 캐시를 미리 채우고 이상치를 알려주는 별도의 배치 작업이 필요했다.

## 아키텍처 결정 — 온디맨드와 배치를 역할로 나누기

두 가지를 같은 무게로 구현하는 대신, 온디맨드가 "실제 기능"을 담당하고 배치가 "온디맨드가 구조적으로 할 수 없는 일"만 보완하는 형태로 역할을 나눴다.

- **온디맨드(배포된 백엔드)**: 의사가 리포트를 열면 그 자리에서 계산. 배치가 꺼져 있어도 사이트는 완전히 정상 동작한다.
- **배치(로컬 Airflow)**: 매일 한 번, 전체 활성 환자를 선제적으로 스캔해서 캐시를 채우고 이상치가 있으면 알림을 보낸다.

배치 오케스트레이션 도구로 관리형 서비스 대신 로컬 docker-compose로 띄우는 Airflow standalone(단일 컨테이너, SequentialExecutor)을 선택했다. 이 프로젝트 규모에서는 실제로 필요한 것이 "DAG 개념을 실제로 운용해 본 경험"이지 대규모 분산 스케줄링이 아니었기 때문에, 비용 없이 그 목적을 달성할 수 있는 최소 구성으로 충분하다고 판단했다.

## 스코프를 좁힌 이유

배치로 옮길 후보가 여러 개 있었지만(예: 외부 문헌 인덱싱, 정기 테스트 검증), 확인해 보니 이런 작업들은 이미 클라우드 스케줄러와 별도 Job으로 안정적으로 운영되고 있었다. 같은 일을 Airflow로 다시 구현하는 것은 중복 작업이라고 판단해서, 이번 배치 파이프라인은 캐시 롤업 DAG 하나와 그에 딸린 알림 하나로 스코프를 좁혔다. 이미 잘 동작하는 것을 굳이 새 도구로 다시 만들지 않는 것도 하나의 선택이라는 판단이었다.

## 구현

롤업 DAG는 매일 한 번 실행되며 두 개의 태스크로 구성된다.

1. **전체 환자 롤업** — 활성 환자 전원(담당 배정 여부와 무관하게 전수 스캔)을 순회하며 각 환자의 최근 기록을 가져와 분석 엔진을 돌리고, 결과를 캐시 테이블에 반영한다. 환자 한 명 처리에 실패해도 그 환자만 건너뛰고 나머지는 계속 진행하도록 개별적으로 예외 처리했다 — 배치 작업 전체가 한 사람의 이상 데이터 때문에 멈추면 안 되기 때문이다. 이상치가 발견된 환자 목록은 다음 태스크로 전달된다.
2. **알림 전송** — 전달받은 목록으로 메시지를 구성해 채팅 웹훅으로 전송한다. 웹훅이 설정되어 있지 않으면 전송을 생략하고 로그만 남기며, 이 경우에도 DAG 자체는 실패로 처리하지 않는다. 알림은 부가 기능이지 핵심 기능이 아니기 때문이다.

새로운 데이터베이스 마이그레이션은 필요하지 않았다 — 캐시 테이블과 이상치 조회용 인덱스는 온디맨드 서비스를 만들 때 이미 "나중에 배치가 조회할 것"을 염두에 두고 만들어 둔 것을 그대로 재사용했다.

## 코드 중복이라는 트레이드오프

분석 엔진 로직은 이미 두 곳(AI 서버, 백엔드)에 복사되어 있었는데, 이번에 Airflow DAG에도 같은 로직이 필요해지면서 세 번째 Python 사본이 생겼다(Kotlin 포팅본까지 합치면 네 번째 언어 사본). 공유 패키지 레포로 분리하는 방법도 검토했지만, 순수 함수 몇백 줄을 공유하자고 별도 배포 파이프라인이 딸린 레포를 하나 더 늘리는 것은 과도한 추상화라고 판단해 기각했다. 대신 코드 중복은 그대로 두되, 합성 데이터로 원본과 사본의 출력이 완전히 일치하는지 직접 확인하는 방식으로 어긋남 위험을 관리했다. 완벽한 해법은 아니지만, 레포 개수와 유지보수 부담 사이에서 실용적인 선택이었다.

## 실행 중 겪은 인프라 이슈

- 로컬에 컨테이너 실행 환경이 아예 없어서 새로 설치해야 했고, 설치 과정에서 운영체제 레벨의 전제 조건(가상화 계층)까지 함께 설치해야 하는 상황이 있었다.
- 알림용 웹훅을 발급하는 과정에서 채팅 플랫폼 쪽 앱 설정이 한 단계 빠져 있어 오류가 났는데, 원인은 앱의 기본 정보(표시 이름) 설정이 누락된 것이었다. 흔히 겪는 설정 순서 문제였고, 해당 항목을 채워 넣자 바로 해결됐다.

## 검증

격리된 개발 환경에서는 컨테이너 실행 권한이 없어 실제로 DAG가 끝까지 도는지 확인할 수 없었기 때문에, 문법 검사와 합성 데이터 기반 출력 비교까지만 그 환경에서 검증했다. 실제 배치 실행은 로컬 PC에서 별도로 확인했다 — 활성 환자 전원을 스캔해 일부 환자에서 이상치를 정상적으로 감지하고, 캐시 테이블에 반영하고, 알림 메시지가 실제로 수신되는 것까지 전 과정이 의도한 대로 동작했다. 정기 스케줄에 의한 자동 실행도 별도 트리거 없이 그다음 날 정상적으로 한 번 더 실행됐다.

## 배운 점

온디맨드와 배치를 "둘 다 완전히 구현"하는 대신 "각자 잘하는 일만 하도록" 역할을 나눈 것이 이 작업에서 가장 중요한 설계 결정이었다. 배치가 없어도 서비스는 완전하게 동작해야 한다는 원칙을 지키면, 배치 쪽 인프라가 아직 불안정하거나(로컬 환경 의존, 컨테이너 미설치 등) 검증이 부족해도 서비스 전체의 신뢰성에 영향을 주지 않는다. 또한 "이미 있는 것을 새 도구로 다시 만들지 않는다"는 판단은 새로운 기술을 도입할 때도 기존 시스템의 실제 운영 상태를 먼저 확인하는 습관이 필요하다는 것을 보여준다.
